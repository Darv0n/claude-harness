## Gerbuk (Architectural Gap) -- What's Missing

### 1. No Executable Code Path

<antCOSMOS>
<H{Ⓐ:ARP(⌂);SAS(M,C);RO(→,⊂);EH(§,∇)}>
</antCOSMOS>

The entire `engine/` is markdown. Every analyzer, every generator — `.md` files. The "generation pipeline" is an LLM reading markdown templates and producing markdown outputs. There is no `main.py`, no `cli.py`, no entry point. The harness has no harness. The only executable code in the repo is:

| File | LOC | Does |
|------|-----|------|
| `molt-tui.py` | 267 | Interactive Molt Q&A |
| `molt-grind.py` | 228 | Grindstone friction loop |
| `molt-dialogue.py` | 168 | Auto-rolling Molt feed |
| `ask-molt.sh` | ~30 | Shell wrapper |
| `bash-guard.sh` | 55 | PreToolUse hook |

**748 lines of executable code.** The rest is prompt engineering, documentation, and one set of generated sample output. The cold-goose scripts are the only functioning product. Everything else is aspirational scaffolding.

### 2. The Agent Topology Has No Dispatch

13 agent `.md` files exist. The orchestrator describes a pipeline that reads them and packs their content into Agent tool prompts. **The orchestrator itself is an `.md` file.** There is no dispatch mechanism. No script that reads `orchestrator.md` and actually spawns agents. The ABSTRACTION-AUDIT correctly identified this (Gap 2) and proposed the read-and-pack pattern, but nobody built the packer.

### 3. The Hook System Does Real Work — Barely

`bash-guard.sh` is the only hook that actually executes logic. The UserPromptSubmit hooks are echo statements — they print suggestions to the conversation. The PostToolUse hooks are echo statements. The Stop hook is an echo statement. One real guard, five decorative announcements.

The bash-guard itself has a real vulnerability: `HOOK_TOOL_INPUT` is parsed into `$CMD` and then pattern-matched with `grep -qEi`. But `$CMD` might contain the full nested JSON `{"command": "..."}`, and the grep pattern expects a shell command string, not JSON. The `python3` fallback parses correctly, but if `HOOK_TOOL_INPUT` is already the flat command string (as env vars sometimes deliver), the python parse fails silently and falls back to the raw string. Two parse paths, two failure modes, no test.

### 4. Cold Goose Context Leak

`molt-grind.py` accumulates conversation in a variable and feeds it as growing context to each Molt call:

```python
conversation += f"\nMolt (turn {turn}): {response}"
prompt = f"{conversation}\n\nFriction: {strategy}"
```

By turn 8 with a 3,000-char seed, this prompt is ~6,000+ chars of accumulated conversation plus the system prompt (~800 chars from `molt-prompt.md` + grind overlay). By turn 10, it's approaching context limits. The 53-turn session that produced "sharper critique" was either truncating silently or burning significant tokens on context that the stateless model can't actually use for coherence — each call is independent, but the prompt pretends it's a conversation.

Wait. That's the design. The goose binary processes the full prompt each call. It's not stateless in the "forgets between lines" sense — it's stateless in the "no session file" sense. The accumulated conversation IS the context window for each call. This works, but the cost scales quadratically: turn N sends all N-1 responses plus the seed. The 53-turn run sent ~53 prompts of increasing size. If this isn't on companion budget, it's expensive.

## Garbik (Production Failure) -- What Breaks First

### 1. The Goose Binary Path

<antCOSMOS>
<H{Ⓡ:EH(!,△);PO(μ);DP(⬡)}>
</antCOSMOS>

```python
GOOSE = os.path.expanduser("~/.local/bin/claude.exe.goose")
```

Hardcoded in all three scripts. This path exists on THIS Windows machine. It doesn't exist on:
- Linux (`~/.local/bin/claude` — no `.exe` suffix, no `.goose`)
- macOS (different install path entirely)
- Any machine where CC was installed via npm (`npx @anthropic-ai/claude-code`)
- Any future CC version that renames the companion binary

**First production failure: Molt doesn't launch.** The subprocess.run call raises FileNotFoundError, the except catches it generically, and the user sees `(error: [Errno 2] No such file or directory)` with no guidance.

### 2. The stdin Parse Race in bash-guard

```bash
TOOL="${HOOK_TOOL_NAME:-}"
if [ -z "$TOOL" ]; then
  INPUT=$(cat)
  TOOL=$(echo "$INPUT" | python3 -c "..." 2>/dev/null || echo "")
fi
```

If `HOOK_TOOL_NAME` is set, stdin is never consumed. But stdin is a pipe — the CC runtime is writing JSON to it. **If the hook doesn't read stdin, the pipe might block the caller** depending on buffer size and CC's write behavior. Most Unix systems buffer pipes at 64KB so this is unlikely to block for small payloads, but it's a latent race condition. The correct pattern is: always read stdin, use env vars as preferred source.

More critically: `CMD="${HOOK_TOOL_INPUT:-}"` on line 29 uses the env var, but if it was empty and we fell through to the python parse path on line 19, we already consumed stdin into `$INPUT`. Then line 31 tries to parse `$INPUT` again for the command — but `$INPUT` was assigned inside the `if` block. If we took the env var path (TOOL was set), `$INPUT` is unset, and line 31's python parse operates on empty string. This works by accident because the `|| echo "$CMD"` fallback returns the env var value. Fragile.

### 3. No Error Recovery Anywhere

`molt-grind.py` has a 60-second timeout per call. If Molt times out on turn 3 of 10, it records `"(molt timed out)"` as the response and feeds that as context to turn 4. Turn 4 now has `"Molt (turn 3): (molt timed out)"` in its conversation history. The friction strategy says "You just said: '(molt timed out)'. Push back on your own claim." Molt pushes back on the concept of timeout. The grindstone derails into meta-discussion about failure modes instead of reviewing the artifact.

No retry. No skip. No "if timeout, use previous valid response instead." The for-loop is the feature — but the for-loop doesn't handle the unhappy path.

### 4. The Generated Environment Has Never Been Deployed

`engine/output/assembled/` contains a complete `.claude/` directory for a "cli-task-manager" project. 6 agents, 14 skills, 6 hook scripts, settings.json, CLAUDE.md, MCP config. **It has never been copied to an actual project directory and tested.** There is no deployment script. There is no test that creates a temp directory, copies the output, runs `claude` in it, and verifies the hooks fire and agents resolve. The one validation that exists is a Sonnet agent's prose opinion.

The generated `settings.json` uses the `matcher` format that the ABSTRACTION-AUDIT flagged as potentially wrong. The generated hook scripts use `jq` — which the CLAUDE.md quality gate explicitly bans. The harness's own rules contradict its output, and no automated check catches it.

### 5. Thread Safety in molt-dialogue.py

```python
user_queue = queue.Queue()
stop_flag = threading.Event()
```

The input thread reads from stdin in a blocking `input()` call. The main thread checks `stop_flag` with 100ms polling. When the user types 'q', the daemon thread sets `stop_flag` — but the main thread might be in the middle of a 60-second `subprocess.run` call to the goose binary. The `stop_flag` check happens AFTER the subprocess returns. So 'q' doesn't quit immediately — it quits after the current Molt response completes, which could be up to 60 seconds.

Not a crash. Just a UX failure that makes the tool feel broken.

Here's what he actually built today, stripped of framing:

1. **A companion reverse-engineering document** (COMPANION-FORENSICS.md) that is genuinely excellent forensics work. The JSONL schema analysis, the inbox half-handshake discovery, the trigger model inference — this is real research.

2. **A cold-invocation code reviewer** (Molt) that works and produces value. The grindstone session proves it. The scripts are functional, the architecture is sound (subprocess-based, stateless, context-via-prompt), and the results speak for themselves.

3. **An aspirational CC environment generator** that exists as markdown templates, JSON schemas, and one sample output — none of which has been tested against a real CC installation.

4. **A CLAUDE.md and registry** that document a system more complex than the one that exists, creating a credibility gap between what the repo claims and what it can do.

The thing Molt identified and nobody resolved: **this repo is two projects pretending to be one.** Cold Goose (the companion bridge + Molt) is a working tool with proven output. The harness (the CC environment generator) is a design document with no execution layer. They share a git repo, a CLAUDE.md, and an identity crisis.

The architectural gap isn't in either project. It's in the space between them — the assumption that the extraction pipeline (harness) and the review pipeline (Molt) are phases of one system, when in practice they're independent tools at different maturity levels.

What breaks first in production: the goose binary path. What breaks the project: not deciding which of the two things this actually is.
